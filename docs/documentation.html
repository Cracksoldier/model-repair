model-repair Documentation

model-repair is a 3D mesh repair tool for Linux and Windows built on CGAL. It automatically detects and fixes the seven most common mesh defects that prevent 3D printing, boolean operations, and FEM analysis from working correctly.

The tool ships as a shared library (libmodelrepair.so / libmodelrepair.dll) with two frontends: a CLI for scripting and a Qt 6 GUI for interactive use. All repair logic lives in the library — the frontends are thin wrappers.

The Repair Pipeline

Steps run in a fixed order. Each step is a precondition for the next: you cannot reliably detect non-manifold vertices in a soup of unconnected triangles, and hole-filling on a mesh with degenerate faces produces garbage. Skipping a step via a flag propagates through all later steps.

1
Merge Duplicate VerticesConvert raw triangle soup → shared-vertex half-edge mesh
2
Remove Degenerate TrianglesDelete zero-area faces before topology analysis
3
Fix Non-Manifold GeometrySplit vertices where more than two faces share an edge
4

5
Fill HolesTriangulate open boundary cycles
6

7

All steps enabled by default. Disable individual steps with --no-<step> flags on the CLI, or by unchecking options in the GUI. See the CLI Reference for the full list.

Step 1 — Merge Duplicate Vertices

1

Merge Duplicate Vertices

Always runs first — required for all subsequent steps
CGAL EPICK

What it fixes

Every common mesh format stores triangles as independent triplets of points. An STL file for a 1000-triangle sphere contains exactly 3 000 point records — one per triangle corner — even though most of those points are geometrically identical. There is no vertex sharing: each triangle is completely isolated from its neighbours.

CGAL's half-edge data structure (Surface_mesh) requires a proper shared-vertex representation where adjacent triangles reference the same vertex objects. Without this, there are no topological edges, no notion of "which faces surround this vertex", and none of the later repair steps can function.

This step also removes degenerate polygons that appear during merging: if two corners of a triangle turn out to be the same point after merging, the triangle has fewer than three distinct vertices and is discarded. Duplicate polygons (identical vertex sets) are also eliminated.

How it works

The implementation uses three CGAL polygon-soup utilities in sequence:

  1. PMP::repair_polygon_soup() — merges points within merge_tolerance of each other and removes degenerate/duplicate polygons. This is the core merging step.
  2. PMP::orient_polygon_soup() — propagates a consistent winding order across the soup. Some faces may already disagree; this makes them consistent before constructing the half-edge structure.
  3. PMP::polygon_soup_to_polygon_mesh() — constructs the CGAL::Surface_mesh half-edge data structure from the now-clean soup.
STL files always report merges. Because STL has no vertex sharing at all, every triangle contributes 3 new points. A clean, correct STL sphere with 1 000 triangles will show ~2 000 merges — this is normal, not a sign of a broken file.

issues_fixed = number of points eliminated (input count minus final vertex count).

Step 2 — Remove Degenerate Triangles

2

Remove Degenerate Triangles

Requires step 1 (shared-vertex mesh)
CGAL EPICK

What it fixes

A degenerate triangle is a face whose three vertices are collinear or effectively coincident — the face has zero area and no well-defined normal. These arise from:

  • Floating-point rounding during boolean operations in CAD tools
  • Very short edges where two vertices nearly coincide but fall just outside the merge tolerance
  • Sliver triangles produced by meshing algorithms at sharp features

Degenerate faces corrupt every subsequent analysis step. Non-manifold detection counts face-edge incidences, and a zero-area face contributes incorrect incidences. Normal orientation propagates winding across faces, and a zero-area face has no valid normal to propagate from. Hole-filling traverses boundary halfedges and may loop indefinitely if the topology is corrupted by degenerate geometry.

How it works

CGAL's PMP::is_degenerate_triangle_face() tests each face individually (checks that the cross product of two edge vectors has non-zero length). All faces that return true are collected, then PMP::remove_degenerate_faces() removes them and re-stitches the surrounding mesh to maintain a valid half-edge structure.

Options

RepairOptions fieldCLI flagTypeDefaultDescription
remove_degenerate_triangles--no-remove-degeneratebooltrueEnable/disable this step.

Report output

issues_found = number of degenerate faces detected.
issues_fixed = number of faces successfully removed.

Step 3 — Fix Non-Manifold Geometry

3

Fix Non-Manifold Geometry

Requires steps 1–2
CGAL EPICK

What it fixes

A manifold mesh is one where the neighbourhood of every point looks like a flat disk — every edge is shared by exactly two faces, and every vertex has exactly one "fan" of faces surrounding it. Many repair and slicing algorithms require this.

Non-manifold geometry occurs in two forms:

  • Non-manifold edges — an edge shared by three or more faces. Common when two separate surfaces are "glued" at a seam, or when a CAD tool exports shared internal walls.
  • Non-manifold vertices — a vertex where two otherwise separate surface patches meet at a single point (a "bowtie" or "pinch point"). The faces around the vertex do not form a single connected disk.

How it works

CGAL's PMP::non_manifold_vertices() (CGAL 6 API) returns a list of halfedge_descriptor values — one per problematic vertex fan. Note: in CGAL 5.x this function returned vertex_descriptor; the signature changed in CGAL 6.

PMP::duplicate_non_manifold_vertices() then resolves the problem by splitting each non-manifold vertex into multiple copies — one per connected fan of faces. The faces retain their geometry; only the topology changes. This may increase the vertex count slightly and may open small seams that the hole-filling step (step 5) will close.

Options

RepairOptions fieldCLI flagTypeDefaultDescription
fix_non_manifold--no-fix-non-manifoldbooltrueEnable/disable this step.

Report output

issues_found = number of non-manifold halfedges detected.
issues_fixed = number of vertices that were split.

Step 4 — Fix Face Normals

4

Fix Face Normals

Requires steps 1–3
CGAL EPICK

What it fixes

A triangle's normal is determined by the order of its three vertices (winding order). Counter-clockwise order (when viewed from outside) points the normal outward. Clockwise order points the normal inward — the face appears "inside-out".

Inconsistent normals cause visible rendering artefacts (faces disappear when viewed from one side), incorrect volume calculations, and failed slicer analysis. They commonly arise when different CAD tools use different winding conventions and objects are assembled without normalising conventions.

How it works

CGAL's PMP::orient() propagates a consistent winding order across each connected component of the mesh by treating the mesh as a graph and doing a BFS/DFS flip propagation. Adjacent faces are flipped so their shared edge is traversed in opposite directions — the standard condition for consistent orientation.

After orientation, the step calls PMP::reverse_face_orientations() on any component where PMP::is_outward_oriented() returns false — but only for closed meshes. That CGAL function asserts the mesh is closed (no boundary edges); calling it on a mesh with open holes would crash. Open meshes skip the outward-check and only receive the consistency pass.

Open meshes get consistency, not outward orientation. If the mesh still has holes after step 3, normals are made consistent across each component but the "which side is outside" check is skipped. Run step 5 (fill holes) to close the mesh before this matters.

Options

RepairOptions fieldCLI flagTypeDefaultDescription
fix_normals--no-fix-normalsbooltrueEnable/disable this step.

Report output

issues_found = 1 if any component needed orientation, 0 otherwise.
issues_fixed = same value (the operation is all-or-nothing per component).

Step 5 — Fill Holes

5

Fill Holes

Requires steps 1–4
CGAL EPICK

What it fixes

An open mesh has boundary edges — halfedges with no opposite face. These appear as "holes" in the surface. They arise from:

  • Exported CAD assemblies where internal walls were deleted to reduce polygon count
  • Boolean operations that left small gaps at seams
  • Manual mesh editing that removed faces without closing the boundary
  • Non-manifold splitting in step 3 pulling apart previously "pinched" surface patches

An open mesh cannot be used for 3D printing (slicers need a watertight shell), volumetric analysis, or many boolean operations.

How it works

The step first identifies all distinct boundary cycles by traversing border halfedges with a visited-set: starting from each unvisited border halfedge, it follows next() calls around the boundary until it returns to the start, recording one "hole" per complete cycle.

Each hole is then filled using one of two CGAL functions:

  • Smooth fill (default, fill_holes_smooth = true): PMP::triangulate_and_refine_hole() — triangulates the hole and then refines the new triangles to match the local mesh density and curvature. Produces higher-quality, smoother patches.
  • Flat fan fill (fill_holes_smooth = false, CLI: --flat-fill): PMP::triangulate_hole() — simple fan triangulation from the boundary centroid. Faster but can produce flat, low-quality patches on curved surfaces.

The max_hole_edges option lets you skip large holes. A hole with more boundary edges than this threshold is left open. Set to 0 (default) to fill all holes regardless of size.

Use max_hole_edges to protect intentional openings. If your model has a deliberate large opening (e.g., a cup with an open top), set --max-hole-edges 50 to fill only small accidental gaps while leaving large boundaries intact.

issues_fixed = number of holes successfully filled.

Step 6 — Remove Self-Intersections

6

Remove Self-Intersections

Requires steps 1–5 · Slowest step
CGAL EPECK

What it fixes

A self-intersecting mesh is one where triangles from different parts of the surface pass through each other. The mesh looks correct visually at a distance but is geometrically invalid: the surface is not a proper 2-manifold embedded in 3D space.

Self-intersections are produced by:

  • Boolean difference operations that don't fully resolve the intersection curve
  • Mesh deformation (sculpting, animation) that folds the surface back on itself
  • Incorrectly assembled CAD parts where components physically overlap
  • Conversion artefacts when importing from formats with lower precision

Slicers, FEM solvers, and volumetric renderers all require a non-self-intersecting surface. Self-intersections cause incorrect infill paths, volume miscalculations, and simulation instability.

How it works

This is the only step that requires CGAL's EPECK kernel (Exact Predicates Exact Constructions). Floating-point arithmetic cannot reliably resolve intersection curves at the precision required — the intersecting triangle pairs must be re-triangulated around the exact intersection line, and floating-point rounding would introduce new intersections in the process.

The algorithm proceeds in four sub-steps:

  1. Detection — PMP::self_intersections() on the EPICK mesh quickly finds all pairs of intersecting faces. If none are found, the step is skipped immediately.
  2. Conversion to EPECK — the EPICK mesh is copied to a CGAL::Surface_mesh<EPECK::Point_3> using CGAL::copy_face_graph(). Coordinates are converted exactly.
  3. Repair — PMP::experimental::remove_self_intersections() with preserve_genus = false. The experimental namespace reflects that this function can change mesh genus (topology) to resolve intersections — it may add or remove handles if that is the only way to make the surface non-self-intersecting.
  4. Conversion back to EPICK — the repaired EPECK mesh is copied back via copy_face_graph() for use in subsequent operations. A final self-intersection check verifies the result.
This step is slow on large meshes. EPECK arithmetic is exact but 10–100× slower than floating-point EPICK. A mesh with 100k faces may take several minutes. Use --no-remove-self-intersections to skip it when you know the mesh has no self-intersections, or when processing many small files where speed matters.

Options

RepairOptions fieldCLI flagTypeDefaultDescription
remove_self_intersections--no-remove-self-intersectionsbooltrueEnable/disable this step. Safe to disable when input is known to be self-intersection-free.

issues_found = number of intersecting face pairs before repair.
issues_fixed = number of pairs resolved. If any remain after repair, they appear as issues_found − issues_fixed.

Step 7 — Remove Internal Geometry

7

Remove Internal Geometry

Requires steps 1–5 (closed mesh) · Disabled by default
CGAL EPICK

What it fixes

Some models contain internal shell fragments — triangles that are completely enclosed inside the outer surface of a closed mesh. These arise from:

  • Boolean operations that leave remnant interior geometry
  • CAD assemblies where internal walls were not removed before export
  • Multi-body models exported as a single file with overlapping volumes

Internal geometry causes incorrect volume calculations, inflated material estimates, and artefacts in slicers that compute infill relative to all visible surfaces. Removing it produces a simpler, lighter mesh representing only the true outer boundary.

Closed mesh required. The inside-test requires a watertight mesh. If the mesh has boundary edges after steps 1–5, this step is skipped automatically and reports faces_removed = 0. Run Fill Holes (step 5) first.

How it works

The step uses CGAL's Side_of_triangle_mesh predicate (EPICK kernel). For each face, the centroid CGAL::centroid(p0, p1, p2) is computed and tested with inside_test(centroid). Faces where the centroid is ON_BOUNDED_SIDE (strictly inside the closed mesh) are collected, then removed via sm.remove_face(f) and sm.collect_garbage().

Options

RepairOptions fieldCLI flagTypeDefaultDescription
remove_internal_geometry--remove-internal-geometryboolfalseEnable/disable this step. Off by default because it is only meaningful on meshes with known interior fragments and is not safe to apply blindly.

Report output

issues_found = number of faces whose centroid was inside the mesh.
issues_fixed = same value (all detected faces are removed).

API

#include "modelrepair/RemoveInternalGeometry.hpp"

// Must be called after repair (mesh must be closed)
modelrepair::RemoveInternalGeometryResult r = modelrepair::remove_internal_geometry(mesh);
std::cout << r.faces_removed << " internal faces removed in "
          << r.duration_ms << " ms\n";

Diagnose Mode

Diagnose mode runs the full detection pipeline on a copy of the mesh and returns a complete RepairReport without modifying the original. Use it to audit a file before committing to a repair, or to compare multiple repair configurations non-destructively.

CLI

Pass --diagnose. OUTPUT is optional — if omitted, no file is written.

# Inspect what would be repaired — no output file
model-repair broken.stl --diagnose --verbose

# Diagnose and write a machine-readable report
model-repair broken.stl --diagnose --report diagnosis.json

The text report header reads === Diagnosis (no changes saved) === instead of the usual repair header.

GUI

Click Diagnose (next to the Repair button). The pipeline runs on a background thread with diagnose_only = true; the report tree fills in exactly as it would after a normal repair. The Save As… button remains disabled — the original mesh on disk is unchanged.

API

using namespace modelrepair;

RepairOptions opts;
opts.diagnose_only = true;
opts.remove_self_intersections = false;   // skip slow step

Mesh mesh = io::load("broken.stl");
const auto v_before = mesh.num_vertices(); // snapshot

RepairReport report = RepairPipeline(opts).run(mesh);

// mesh is unchanged
assert(mesh.num_vertices() == v_before);
assert(report.diagnose_only == true);
Diagnose mode disables smoothing and decimation automatically. Even if opts.smooth = true or opts.decimate = true, neither is applied when diagnose_only = true — the mesh must not be modified.

Mesh Decimation

Decimation reduces the polygon count of the repaired mesh while preserving overall shape. It runs after all seven repair steps and is separate from the pipeline: a clean, watertight mesh is always the intermediate product.

Backends

Three decimation backends are available, selectable in the GUI, Batch window, and API:

CGAL (default) Greedy edge-collapse with Face_count_ratio_stop_predicate — Always available. In-place — preserves all property maps natively. May stop slightly above the target when further collapses would violate manifold constraints.
MeshOptimizer meshopt_simplify() — error-bounded simplification Max error (default 0.01) — the algorithm stops early if the next collapse would exceed this normalized error threshold Often faster than CGAL on large meshes. Intermediate save

When decimation is requested from the CLI, the mesh is automatically saved to an intermediate file immediately before decimation — after any smoothing that was also requested:

#   out_repaired.stl  — repaired, full resolution (post-smooth if --smooth was used)
#   out.stl           — decimated to ~50 % of faces
model-repair broken.stl out.stl --decimate 0.5

# Keep 50 % of faces — CGAL backend (default)

Mesh Smoothing [experimental]

Smoothing repositions mesh vertices to reduce surface roughness without changing topology. It is a post-repair operation: the repair pipeline runs first to produce a clean, watertight mesh, and then smoothing softens its surface. Face and vertex counts are unchanged — only vertex positions move.

Typical use cases: voxel/CT-scan exports with stepped surfaces, 3D-scanned meshes with photogrammetry noise, low-poly CAD with harsh faceting, or hole-fill patches that look jarring on curved surfaces.

How it works

Cotangent-weighted feature-preserving Laplacian (λ = 0.5)

Each iteration moves every interior vertex toward a weighted average of its 1-ring neighbours. The weight for each neighbour is cot(α) + cot(β) — the cotangents of the angles opposite the shared edge in its two adjacent triangles. This is the standard discrete Laplace-Beltrami operator: it correctly reflects the geometry of irregular triangulations and produces visibly less distortion per iteration than uniform averaging.

Feature preservation via soft crease falloff. Each neighbour's weight is additionally scaled by a factor that depends on the dihedral angle of the shared edge. For a perfectly flat edge the scale is 1.0 (full contribution). At the crease angle threshold the scale reaches 0.0 (excluded). Between the two the scale decreases linearly — so edges near the threshold contribute partially rather than being hard-cut. The result: large flat surfaces smooth aggressively, sharp seam edges are preserved as hard barriers, and mild curvature features (facial geometry, subtle costume details) erode significantly more slowly.

Crease angle guide: 20–30° = preserve almost all curvature, minimal smoothing of soft features; 45° (default) = good balance for organic meshes; 80°+ = smooth aggressively, only the sharpest seams survive.

Volume preservation. Laplacian smoothing shrinks the mesh slightly each iteration. After all passes, if the mesh is closed (watertight), the vertices are scaled uniformly around their centroid to restore the original volume. The Before/After report will show volume_before ≈ volume_after for most inputs.

Effective only on meshes with surface irregularities. On a well-formed UV sphere or similar parameterised mesh, the vertex positions are already close to their ideal locations. Laplacian smoothing can only move each vertex toward its neighbours' centroid, which barely changes for regular geometry. The visible "faceting" of a coarse sphere (caused by flat shading of large faces) cannot be removed by smoothing alone — combine with Isotropic Remeshing to add geometry first.

Pitfalls

1. Volume shrinkage on open meshes

Volume is automatically restored after smoothing for closed (watertight) meshes. Open meshes do not have a well-defined volume, so shrinkage is not corrected. If dimensional accuracy matters for an open mesh, use the minimum number of iterations and verify the result in the Before/After preview.

2. Sharp feature erosion

The crease angle controls how aggressively sharp edges are protected. At the default 45°, edges with a dihedral angle larger than 45° act as smoothing barriers. Mechanical parts with precise corners (text emboss, gear teeth, chamfers) may still erode if the crease angle is too high — lower it to 20–30° or avoid smoothing altogether for precision parts.

3. Coarse flat surfaces — smoothing alone is not enough

On a low-poly mesh with large flat triangles (e.g. a character body surface exported from a game engine), the flat faceting is caused by too few triangles, not by misplaced vertices. Laplacian smoothing can only move existing vertices; it cannot add the interior vertices needed to represent smooth curvature. Use Isotropic Remeshing before smoothing to subdivide those coarse faces first.

4. Open meshes and boundary drift

Smoothing works on meshes that still have boundary edges (open holes). Boundary vertices are skipped (not moved), so boundary curves are preserved. Interior vertices adjacent to the boundary can still drift slightly. Enable hole filling before smoothing if preserving hole boundaries is important.

5. Iteration count has diminishing returns

The biggest improvement happens in the first 3–5 iterations. Beyond 10, gains in smoothness become small while feature erosion continues to grow. The default of 3 iterations is deliberately conservative — start there and increase only if the result is still too rough.

6. Smooth before decimate, not after

The execution order is: repair → remesh → smooth → decimate. Smoothing a decimated mesh with large, irregular triangles produces poor results because the Laplacian stencil is uneven. Smooth (and remesh if needed) the dense mesh first, then decimate.

CLI

# Smooth 5 iterations, preserve edges sharper than 45°
model-repair blocky.stl smooth.stl --smooth 5

# Smooth with tighter feature preservation (lower crease angle)
model-repair character.stl out.stl --smooth 3 --smooth-crease-angle 25

# GPU-accelerated smoothing — falls back to CPU if Vulkan is unavailable
model-repair blocky.stl smooth.stl --smooth 10 --smooth-vulkan

# Smooth then decimate to 50 % of faces
model-repair blocky.stl out.stl --smooth 3 --decimate 0.5

Iteration count range: 1–50. Crease angle range: 0–180°. Exit code 6 is returned if smoothing fails.

GUI

Check Smooth after repair in the options panel. The Iterations spinbox (1–50, default 3) and Crease angle spinbox (0–180°, default 45°) become active. Smooth and decimate can be combined — smooth runs first, then decimate. Both are automatically disabled in Diagnose mode.

Isotropic Remeshing [experimental]

Remeshing introduces new geometry by redistributing and subdividing the mesh triangles. Unlike smoothing, which only moves existing vertices, remeshing adds new vertices in areas with coarse triangulation — enabling the follow-up Laplacian to represent smooth curvature where there were previously no interior vertices to work with.

Run before smooth, not instead of it. The recommended workflow for low-poly organic meshes is: repair → remesh → smooth → (optional) decimate. Remeshing alone already improves triangle quality; pairing it with a few smooth iterations produces visibly superior results.

How it works

Each remeshing iteration applies four operations in sequence (CGAL PMP::isotropic_remeshing):

  1. Split edges longer than 4/3 × target_len
  2. Collapse edges shorter than 4/5 × target_len
  3. Flip edges to improve vertex valence (toward degree 6)
  4. Relocate vertices toward their Laplacian centroid (tangentially)

The target edge length is mean_edge_length × edge_length_factor. A factor of 1.0 redistributes triangles to a uniform size without changing total count. A factor below 1.0 adds triangles in coarse areas (increasing count); above 1.0 coarsens the mesh. The default of 0.8 adds roughly 30–60 % more triangles on typical low-poly inputs.

Selective: coarse faces only

Only faces with at least one edge longer than target_len are passed to the remesher. Fine-geometry regions (face, hands, fingers) that already have edges shorter than the mean are automatically excluded and left untouched. CGAL protects the boundary between selected and unselected regions, so the mesh remains watertight at the transition.

This means you can use a moderate factor (0.7–0.9) on a character mesh and the coarse body surfaces are subdivided while facial detail is fully preserved.

Normal Map Displacement [highly experimental]

Normal Map Displacement bakes surface detail from a tangent-space normal map image into real displaced geometry. This is useful when a character model or prop was designed to rely on a normal-map shader for surface detail — displacement converts that shader detail into actual printable geometry.

Highly experimental. Results depend strongly on UV quality and normal map type. The pseudo-height derivation is a heuristic; for production use a proper height-map bake from your DCC tool is more accurate. Only OBJ files with complete UV data are currently supported.

How it works

  1. Pre-subdivide — run N passes of Loop subdivision to increase vertex density so there are enough vertices to capture normal-map detail. UV coordinates are propagated to new vertices automatically.
  2. Sample the normal map — for each vertex, look up the pixel at the vertex's UV coordinate (nearest-neighbour, with flip_y_uv to correct for image vs UV Y-axis orientation).
  3. Decode pseudo-height — the blue/Z channel encodes how "tilted" the surface is. After decoding to [-1, 1]: 
    pseudo_height = (1.0 − nz) × 0.5
    Flat regions (nz ≈ 1) → pseudo_height ≈ 0 (no displacement). Maximally tilted regions (nz ≈ −1) → pseudo_height ≈ 1 (full displacement_strength push).
  4. Displace — move each vertex outward along its geometric normal by pseudo_height × displacement_strength mm.
UV coordinates required. The mesh must have a "v:uv" vertex property map. Load an OBJ file exported from Blender/Maya/etc. with UV data included. STL and 3MF files have no UV support. The displacement accordion in Wizard Phase 2 is disabled and shows a warning when the loaded mesh has no UV map.

Parameters

displacement_strengthfloat0.3Maximum outward displacement in mesh units (mm). Flat surfaces are not displaced; maximally tilted surfaces are pushed by this full amount.
pre_subdivisionsint2Number of Loop subdivision passes to run before displacing (0–4). Each pass quadruples the face count.
flip_greenboolfalseInvert the green/Y channel. Enable for DirectX-convention normal maps (Y points down); leave off for OpenGL convention (Y points up).
flip_y_uvbooltrueFlip the V coordinate before sampling (fv = 1 − fv). Required because image Y=0 is the top row while UV Y=0 is the bottom.

Normal Map → Height Map Converter

The Normal Map → Displacement Map tool converts a tangent-space normal map image into a physically correct grayscale height map using Poisson reconstruction of the gradient field. Unlike the mesh-based Normal Map Displacement feature, this tool is a standalone image-in / image-out operation — no mesh is required. The output is a 16-bit PNG suitable for use as a displacement map in a renderer, sculpting application, or CNC milling workflow.

Where to find it. Open the Tools dropdown in the toolbar at the top of the main window and select Normal Map → Displacement. The tool opens as a separate non-modal window; you can keep it open alongside the main repair workflow.

How it works

  1. Load & decode — the normal map is loaded with stb_image and each RGB pixel is decoded to a unit normal vector (nx, ny, nz) ∈ [−1, 1]³. Enabling Flip green channel negates ny for DirectX-convention maps.
  2. Build gradient field — the surface gradient of the height map is derived from the tangent normals: 
    gx = −nx / max(nz, ε) × gradient_strength
gy = −ny / max(nz, ε) × gradient_strength
    Values are clamped to ±20 to suppress artefacts from near-horizontal normals.
  3. Compute divergence — backward finite differences give ∂gx/∂x + ∂gy/∂y, which forms the right-hand side of the Poisson equation.
  4. Poisson solve — an N×N sparse Laplacian (5-point stencil, Neumann boundary conditions at image edges, Dirichlet h[0,0] = 0 to anchor the solution) is assembled and solved with Eigen's ConjugateGradient + IncompleteCholesky preconditioner. IC dramatically reduces the effective condition number of the 2D Laplacian compared to a simple Jacobi preconditioner — convergence typically occurs in 50–150 iterations for images up to ~2 MP, versus ~3000 iterations without preconditioning. The status line reports the actual iteration count and whether the solver converged.
  5. Post-process — optional box blur (smooths solver noise), normalize to [0, 1], apply contrast, optionally invert.
  6. Export — the height map is saved as a 16-bit grayscale PNG (QImage::Format_Grayscale16), preserving the full dynamic range.
Image size limit: 8 megapixels (≈ 2828 × 2828). The Poisson solve requires an N × N sparse matrix where N is the pixel count. At 1024 × 1024 (~1 MP) the solve takes a few seconds; at 2048 × 2048 (~4 MP) it may take 30–60 s and needs ~240 MB of RAM. Images larger than 8 MP are rejected with a clear error message — downsample first in an image editor if needed.

Wall Thickness Heatmap

The wall thickness heatmap colours each face of the after-mesh by the measured local wall thickness. Thin walls appear blue; thick regions appear red. This lets you visually identify areas likely to fail during FDM or resin printing before slicing.

How it works

An AABB tree is built over all mesh faces. For each face:

  1. Compute the face centroid.
  2. Cast a ray from centroid + ε × outward_normal inward along −normal.
  3. Find the first intersection via AABB_tree::first_intersection(ray).
  4. The intersection distance is the local wall thickness at that face.

Results are normalised to [0, 1] using the 95th percentile as the maximum cap, then mapped through the heatmap: scalar = 1 − clamp((t − min) / range, 0, 1) so thin = 1 = red, thick = 0 = blue.

How to enable

Overhang Highlight

The overhang highlight colours each face by how steeply it overhangs the print bed. Well-supported horizontal faces appear blue; steep overhangs appear red. This is a quick visual check for areas that may require support structures.

Performance & Parallelism

The mesh smoothing step supports two parallel execution modes that can drastically reduce wall-clock time on large meshes: a multi-core CPU path using Intel TBB and a GPU compute path using Vulkan. Both are opt-in at build time and at runtime — the sequential CPU path is always the fallback.

Only smoothing is parallelized. Isotropic remeshing uses CGAL's PMP::isotropic_remeshing, which is single-threaded in CGAL 6.x — there is no Parallel_tag hook in the current implementation. Decimation uses a greedy edge-collapse algorithm where each collapse changes the priority queue, making meaningful parallelisation impossible. Smoothing is the only step where all vertex updates within one iteration are independent: each vertex reads its 1-ring neighbours (read-only within an iteration) and writes its own output position, with no write conflicts.

TBB CPU Parallel Path

How it works

When compiled with TBB (MODELREPAIR_ENABLE_TBB=ON, the default), the inner vertex-update loop in each Laplacian iteration is dispatched with std::execution::par_unseq over all interior vertices. TBB provides the thread pool and a work-stealing scheduler; the standard library execution policy selects it automatically at link time.

The loop body reads precomputed cotangent weights and neighbour positions — both read-only within an iteration — and writes one output position per vertex to a temporary buffer. After all vertices are updated in parallel the buffer is swapped into the mesh. There is no shared mutable state between threads: the computation is embarrassingly parallel within each iteration.

Vulkan GPU Compute Path

Key advantage: zero per-iteration round-trips

The Vulkan path uploads vertex data to device-local GPU memory once before the first iteration, runs all iterations entirely on the GPU, and downloads the result once after the last iteration. There are no CPU↔GPU data transfers between iterations — the cotangent weights, neighbour indices, and vertex positions all remain in VRAM throughout the entire smooth() call.

This is the primary reason the GPU path outperforms TBB for large iteration counts: TBB requires a write-barrier and buffer swap between every iteration; Vulkan uses a lightweight in-command-buffer pipeline barrier — no data leaves the GPU.

CSR sparse-matrix layout on the GPU

Cotangent weights are stored as a CSR (Compressed Sparse Row) sparse matrix in three device-local buffers:

  • row_offsets[V+1] — uint32: row i spans non-zeros [row_offsets[i], row_offsets[i+1]). Border vertices have empty rows (start == end) and are passed through unchanged.
  • col_indices[nnz] — uint32: the neighbour vertex index for each non-zero entry.
  • weights[nnz] — float: the combined cotangent × crease-falloff weight. Border-vertex rows are empty — no column entries.

This layout gives coalesced sequential access within each workgroup: threads processing consecutive vertex IDs read consecutive rows, and the inner loop over neighbours is a linear scan through col_indices and weights with no random jumps in the row-offset array.

Ping-pong position buffers

Two position buffers (pos_a and pos_b, vec4 per vertex — xyz + one float of padding for alignment) alternate as input and output each iteration. Two pre-configured Vulkan descriptor sets encode this without any rebinding: desc_sets[0] reads pos_a and writes pos_b; desc_sets[1] reads pos_b and writes pos_a. Only the bound descriptor set index changes between iterations — no buffer copies, no layout transitions, no data movement.

After N iterations the final result is in pos_b when N is odd and in pos_a when N is even. The download stage maps and reads the correct buffer.

The GLSL compute shader

The shader (src/shaders/smooth_laplacian.comp) dispatches 256-thread workgroups. Each invocation handles one vertex: reads its row from the CSR buffers, accumulates the weighted-average neighbour position, and applies the Laplacian step (λ = 0.5). Border vertices (empty CSR row) are passed through without modification. 

// smooth_laplacian.comp — key structure (simplified)
layout(local_size_x = 256) in;
layout(std430, binding = 0) readonly  buffer PosIn   { vec4 pos_in[];  };
layout(std430, binding = 1) writeonly buffer PosOut  { vec4 pos_out[]; };
layout(std430, binding = 2) readonly  buffer RowOff  { uint row_off[]; };
layout(std430, binding = 3) readonly  buffer ColIdx  { uint col_idx[]; };
layout(std430, binding = 4) readonly  buffer Weights { float wt[];     };
layout(push_constant) uniform PC { uint vertex_count; float lambda; };

void main() {
    uint vid = gl_GlobalInvocationID.x;
    if (vid >= vertex_count) return;
    uint start = row_off[vid], end = row_off[vid + 1];
    if (start == end) { pos_out[vid] = pos_in[vid]; return; } // border vertex
    vec4 wsum = vec4(0.0); float wtot = 0.0;
    for (uint k = start; k < end; ++k) {
        wsum += wt[k] * pos_in[col_idx[k]]; wtot += wt[k];
    }
    pos_out[vid] = pos_in[vid] + lambda * (wsum / wtot - pos_in[vid]);
}

The shader is compiled from GLSL to SPIR-V by glslc at CMake build time. A small CMake script (cmake/EmbedSpv.cmake) then byte-swaps the SPIR-V binary into a static const uint32_t smooth_laplacian_spv[] C array that is included only by VulkanSmoothing.cpp. There is no runtime shader compilation — the SPIR-V bytecode is baked into the binary at link time.

Weight precomputation approximation

Cotangent weights and crease-falloff factors are computed once from the initial mesh before any iteration, then uploaded to the GPU and held constant for the entire smooth() call. The approximation: crease-falloff factors use the initial face normals rather than per-iteration updated normals.

For typical usage (5–20 iterations, moderate crease angles ≥ 20°) the visible difference from an exact per-iteration computation is negligible — dihedral angles change slowly over the iteration sequence, so the crease weights remain accurate to well within one degree throughout. This tradeoff eliminates all CPU↔GPU round-trips and is the reason the GPU path scales so well to large iteration counts.

If exact per-iteration crease barriers are critical (e.g. very low crease angles on meshes that deform significantly per iteration), use the CPU path.

Software Vulkan implementations are supported but slower. On systems with only a software renderer (lavapipe, SwiftShader), smooth_vulkan_available() returns true — these devices appear as normal Vulkan physical devices. The GPU path will work, but will run on the CPU emulated in software and is likely slower than TBB. Prefer TBB in VM or container environments without a real GPU.

When GPU wins over TBB

The upload/download overhead for the GPU path is approximately constant (~5–20 ms, depending on mesh size and PCIe bandwidth). Breakeven is typically:

  • ≥ 5 iterations on a 100k+ vertex mesh — GPU throughput amortises the upload overhead; each subsequent iteration adds only ~2–5 ms on a mid-range GPU vs ~25 ms/iter for TBB on a 12-core machine.
  • Any iteration count on a 500k+ vertex mesh — even for a single iteration, 500k vertices saturate GPU shader lanes before the overhead matters.

For small meshes (< 10k vertices) or single iterations, TBB wins: GPU overhead exceeds the compute time.

# Diagnose only — no output file written, mesh untouched

# Repair + decimate to 50 % of faces

# Smooth a blocky voxel mesh (5 angle-smoothing iterations)
model-repair blocky.stl smooth.stl --smooth 5

model-repair blocky.stl out.stl --smooth 3 --decimate 0.5
# GPU-accelerated smoothing (falls back to CPU automatically if Vulkan unavailable)
model-repair blocky.stl out.stl --smooth 10 --smooth-vulkan
# Enable step 7 — remove geometry hidden inside the closed mesh

Running a repair

Click Repair. The repair runs on a background thread (RepairWorker on a QThread) so the UI stays responsive. While running, all input controls (options panel, Open file, Batch Repair, drop zone) are disabled to prevent conflicting changes. A progress bar advances one slot per operation across all enabled steps — 7 pipeline steps, then one slot per remesh iteration ("Remeshing 1/3 → 2/3 → 3/3"), one slot per smooth iteration ("Smoothing 1/3 → 2/3 → 3/3"), and one slot for decimate — so the bar moves continuously rather than freezing on a single step. An elapsed time counter in M:SS / M:SS format (total elapsed / current step) on the right updates every second to confirm the tool is still working; the right-hand figure resets to 0 at the start of each new step. The report tree fills in per-step results as each step completes. Execution order: repair → remesh → smooth → decimate.

On success, the 3D Preview Window opens automatically showing the before/after comparison (the "after" mesh is the decimated version when decimation is enabled). On failure, an error message is shown in the status bar.

Diagnose

Click Diagnose to run the pipeline without modifying the mesh. The report tree fills in exactly as after a normal repair, showing what would be changed. The Save As… button is not enabled after a diagnosis run and no preview window opens — the original file on disk is untouched.

Saving the result

Click Save As… to open a save dialog. The output format is determined by the file extension you choose (.stl writes binary STL, .obj writes OBJ, .3mf writes 3MF, .ply writes binary LE PLY, .glb writes binary glTF, .gltf writes text glTF). The saved mesh is the decimated version if decimation was enabled.

Repair Wizard

The Wizard… button opens a guided three-phase workflow. It is enabled whenever a file is loaded and is disabled while a background repair is running in the main window. Only one wizard instance can be open at a time — clicking the button again raises the existing window rather than opening a second one.

Each phase runs on a background thread (WizardWorker on a QThread) so the UI stays responsive. The wizard window cannot be closed while a phase is running. The Cancel button at the bottom remains enabled at all times: during Phase 2 (Remesh & Smooth) it interrupts the operation after the current iteration completes and returns to the options page; during Phase 1 and Phase 3 the current operation runs to completion before returning. Each running state also shows a live M:SS / M:SS elapsed-time counter (total phase / current step) initialised to 0:00 / 0:00 as soon as the phase starts.

After each phase completes, the page switches to a before/after 3D preview with the same camera controls as the 3D Preview Window. Both viewports share a single camera state so they always show the same angle. Clicking Save As… at any phase exit point opens a save dialog — the wizard closes only after a successful save (cancelling the dialog or a save error leaves it open so you can try again or continue to the next phase).

Technical details

The renderer requests only a 24-bit depth buffer — no CoreProfile, no MSAA. This is required for compatibility with the NVIDIA proprietary driver on Wayland (EGL backend), which rejects QSurfaceFormat::CoreProfile with EGL_BAD_ATTRIBUTE. The GLSL 3.30 shaders still work because the default EGL context provides OpenGL 3.3+ in compatibility mode.

File Format Support

STL (Stereolithography)

The most common format for 3D printing. STL stores a flat list of triangles with no vertex sharing and no colour or metadata. Both binary and ASCII variants are supported.

  • Read: Auto-detected (binary vs ASCII) from file content, not extension.
  • Write: Binary by default (smaller file); ASCII with --ascii-stl / io::save(mesh, path, false).
  • Note: Because STL has no vertex sharing, step 1 always reports merges — this is expected.

OBJ (Wavefront)

A text format that supports shared vertices, material references, and UV coordinates. Only triangle geometry is processed; materials are not written on save.

  • Read: Triangulated on load. Quads and higher-order polygons are not supported. Per-vertex RGB color is read when all v lines have the optional r g b suffix (Meshlab / CloudCompare extension); stored as a "v:color" property map. UV coordinates are read from vt u v lines and the /vt slot of face entries (f v/vt/vn); per-corner UVs are averaged into a per-vertex "v:uv" property map (UV2 = std::array<float,2>). The UV map is attached only when every face corner has a UV index and the mesh did not gain extra vertices during polygon-soup orientation. This UV data is required for Normal Map Displacement.
  • Write: Plain triangulated OBJ, no materials. When a "v:color" property map is present, each vertex line is written as v x y z r g b (r/g/b in [0,1]). UV data is not currently written back.

3MF (3D Manufacturing Format)

An XML-based format designed for 3D printing. Supports multiple mesh objects, metadata, colours, and print settings. The library wraps lib3mf (fetched automatically by CMake).

  • Read: Loads the first mesh object from the file. Multi-object 3MF files have only the first object repaired.
  • Write: Writes a single-object 3MF. Metadata and print settings from the input are not preserved.

GLB / glTF 2.0

The binary (.glb) and text (.gltf) variants of glTF 2.0 — the format used by Blender, Bambu Studio, PrusaSlicer, and most web 3D viewers. The library wraps tinygltf v2.8.21 (fetched automatically by CMake).

  • Read: Loads all triangle primitives from all meshes in the file. Non-triangle primitives and non-indexed geometry are skipped. Only geometry is imported; materials, textures, and node transforms are ignored. Per-vertex color is read from the COLOR_0 accessor (VEC3/VEC4, FLOAT/UNSIGNED_BYTE/UNSIGNED_SHORT) when every primitive in the file has it; stored as a "v:color" property map.
  • Write: Always writes binary GLB regardless of the extension passed to write_glb. Vertex positions are stored as VEC3 FLOAT; indices as SCALAR UNSIGNED_INT. When a "v:color" property map is present, a COLOR_0 accessor (VEC3 FLOAT) is included. A single mesh with one primitive is written; no materials or scene metadata beyond the default scene node.
  • Note: stb_image is disabled at compile time — tinygltf is used in geometry-only mode.

PLY (Polygon File Format)

The standard format for 3D scanner and photogrammetry output (Meshlab, CloudCompare, Open3D, COLMAP). Supports ASCII and binary (little-endian and big-endian) variants. CGAL's <CGAL/IO/PLY.h> handles all three transparently — no extra dependency.

  • Read: Parses the polygon soup using CGAL; orients and builds the half-edge mesh. Per-vertex RGB color is read when the PLY header declares red, green, blue vertex properties; stored as a "v:color" property map.
  • Write: Binary little-endian PLY. Vertex data is float x/y/z, with optional uchar red/green/blue appended when a "v:color" property map is present. Faces are stored as [count=3, uint i0, i1, i2].

Pipeline design

The pipeline is designed around a strict ordering guarantee: each step is a hard precondition for the next. The implementation enforces this by making each step a private method of RepairPipeline called in sequence — there is no way to call step 3 without having run steps 1 and 2.

Each step mutates the Mesh in-place and returns a StepReport with counts and timing. The pipeline collects all StepReports into the final RepairReport.

EPICK vs EPECK

CGAL offers multiple arithmetic kernels. Steps 1–5 use EPICK (Exact Predicates Inexact Constructions) — geometric predicates (e.g., "do these two triangles intersect?") are computed exactly, but new point coordinates (e.g., centroids for hole filling) are computed in double-precision floating point. This is fast and accurate enough for steps that only rearrange topology or construct simple patches.

Step 6 requires EPECK (Exact Predicates Exact Constructions). Resolving a self-intersection means computing the exact intersection line between two triangles and re-triangulating both around it. If this is done in floating point, the new vertices introduce new (slightly offset) intersections, and the repair loops forever. EPECK uses lazy exact arithmetic (GMP/MPFR under the hood) to compute intersection coordinates to arbitrary precision — but this is 10–100× slower than floating-point arithmetic.

GUI threading model

The GUI runs the repair on a background thread to keep the UI responsive. The design:

  • RepairWorker is a QObject moved onto a QThread. It holds a copy of the input mesh (taken before RepairPipeline::run() mutates it in-place) and the options struct.
  • Progress signals cross the thread boundary via QMetaObject::invokeMethod(..., Qt::QueuedConnection) — the only thread-safe way to update Qt widgets from a non-GUI thread.
  • On completion, RepairWorker emits a four-argument finished signal carrying: RepairReport, before-mesh, after-mesh, error string. MainWindow receives this on the main thread and opens the PreviewWindow.

Camera synchronisation in PreviewWindow

Both MeshViewWidget instances (Before and After) share a single std::shared_ptr<CameraState> containing rotation (as a QQuaternion), zoom, pan, and scene radius. When any camera-mutating mouse event fires, the widget calls emit camera_changed(); the peer widget is connected to that signal and calls update() to redraw. No mutex is needed because both widgets live in the main thread — Qt's event loop is single-threaded.

Linux — AppImage Release

Pre-built Linux AppImages are published on the GitHub Releases page. An AppImage is a self-contained executable that bundles Qt 6, libmodelrepair, lib3mf, and all other runtime dependencies. It runs on any x86-64 Linux without installation or root access.

Running the AppImage

# Set the version tag from the Releases page, then download
VERSION=v0.6.0
wget https://github.com/Cracksoldier/model-repair/releases/download/${VERSION}/model-repair-${VERSION}-x86_64.AppImage
chmod +x model-repair-${VERSION}-x86_64.AppImage

# Launch the GUI
./model-repair-${VERSION}-x86_64.AppImage

# Access the CLI inside the AppImage
./model-repair-${VERSION}-x86_64.AppImage --appimage-extract-and-run \
  broken.stl repaired.stl --verbose

# Or extract all contents and use the binaries directly
./model-repair-${VERSION}-x86_64.AppImage --appimage-extract
./squashfs-root/usr/bin/model-repair broken.stl repaired.stl
FUSE is not required. The AppImage uses APPIMAGE_EXTRACT_AND_RUN=1 internally, so it self-extracts to a temporary directory without needing a FUSE mount. This works in containers and restricted environments.

Version Bump

If next_version is provided, the workflow performs these steps after the release is created:

  1. Updates the VERSION field in CMakeLists.txt (e.g. 0.1.0 → 0.2.0).
  2. Commits the change as github-actions[bot].
  3. Pushes the commit to the branch that triggered the workflow (usually main).

The released tag (e.g. v0.1.0) remains pinned to the pre-bump commit — the exact source that was used to build the AppImage. The main branch then moves forward to the development version.

Releases are created as drafts. After the workflow succeeds, go to GitHub Releases, review the draft, add release notes, and publish it. The AppImage artifact is already attached.
